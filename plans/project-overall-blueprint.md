# 法律证据智能解析平台 · 整体项目蓝图

> 版本：V2.2  
> 适用范围：Web 端 + 微信小程序端（律师/机构/当事人）+ 统一后端（本地开发 / 腾讯云分阶段生产）

---

## 1. 项目目标与定位

本项目面向法律行业，构建一套 SaaS 协同办案平台，为律师与当事人提供案件管理、证据上传、AI 解析、报告下载的一体化能力。

核心目标：

- 律师端：提升案件处理效率，统一管理案件、当事人、证据与 AI 分析结果。
- 当事人端：通过极简页面上传证据、查看解析进度与结果、下载报告。
- 平台端：通过多租户与权限体系保障数据隔离，支持后续按量收费。

---

## 2. 一体化原则（Web 与小程序）

### 2.1 数据一致性

- Web 与小程序共享同一套后端 API。
- 案件、文件、解析记录、时间流统一存储在腾讯云数据库。
- 律师端和当事人端展示同源数据，不做双写，不做端侧状态源。

### 2.2 权限隔离

- 接口层通过 `tenant_id` + `user_id` 进行鉴权和数据过滤。
- 当事人端仅能访问与自己关联的案件，默认只读。
- 当事人可执行操作仅包括：上传补充材料、查看可见内容、下载角色可见 PDF。

### 2.3 实时同步

- 律师端操作（上传、重解析、修改案件信息）实时反馈到当事人端。
- 推送策略：优先 WebSocket；无法建立长连接时降级为轮询。

## 3. 总体架构蓝图

### 3.1 技术栈建议

| 层级 | 技术选型 | 说明 |
| --- | --- | --- |
| Web 前端 | Vue 3 + Element Plus + Vite | 律师/机构后台，按角色显示差异化导航 |
| 小程序前端 | uni-app + 微信小程序 + TDesign | 与 Web 同构的律师/管理员工作台，额外承担当事人链路 |
| 后端 | FastAPI + SQLAlchemy + Pydantic | 统一 API、认证、AI 任务与多租户权限 |
| 数据库 | PostgreSQL 15 / TencentDB PostgreSQL | 开发可本地，生产以托管数据库为准 |
| 缓存 | TencentDB Redis | 会话、缓存、限流、任务状态 |
| 对象存储 | 腾讯云 COS + CDN | 证据文件、报告文件、导出文件与云端分发 |
| 异步任务队列 | TDMQ/CMQ + Worker | 生产目标队列；开发/过渡环境允许保留 DB Queue |
| AI 服务 | OCR + ASR + 大模型 + RAG | 证据提取与总结 |
| 部署 | 第一阶段：`CVM + Docker Compose + COS + 可选 CDN`；第二阶段：`CLB + 多 CVM` 或 `TCR + TKE + WAF + CDN` | 当前推荐先以 `CVM + COS` 完成商用上线，后续再按租户数、流量和运维复杂度扩容 |

### 3.2 服务划分

- 用户服务：认证、授权、租户与角色。
- 案件服务：案件信息、时间流、文件元数据。
- AI 服务：解析任务、摘要生成、缺失证据识别、PDF 报告。
- 文件服务：COS 上传签名、回调与预览控制。
- 通知服务：WebSocket 推送、短信与站内通知。
- 计费服务：AI 用量统计与套餐扣减。

### 3.3 当前推荐商用拓扑（CVM + COS）

```text
用户 / 小程序
    |
域名 + TLS
    |
DNSPod -> Nginx（CVM）
    |
├── Web Frontend
├── Backend API
├── AI Worker
└── Report Service
    |
    +--> PostgreSQL（同机或托管）
    +--> Redis（同机或托管）
    +--> COS（案件文件 / 报告文件）
    +--> 可选 CDN（下载与静态加速）
```

当前推荐商用原则：

- 第一阶段允许使用 `单台 CVM + Docker Compose` 快速上线，但服务必须在云端独立启动和运行，不依赖开发机、办公室网络或家宽公网。
- 文件与报告必须进入 `COS`，不再继续以应用本地卷作为长期生产真源。
- `CVM` 建议采用按流量计费，仅承载 API、页面和少量动态响应；附件、报告与静态下载优先通过 `COS` 或 `COS + CDN` 分发。
- 预算有限时可先将 `PostgreSQL/Redis` 放在同一台 `CVM`；若计划更稳妥的生产运维，则优先切换到托管数据库与托管 Redis。
- 该形态适合单律所和首批多租户商用，不作为长期高可用终态。

### 3.4 演进目标拓扑（云端托管终态）

```text
用户 / 小程序
    |
域名 + TLS
    |
WAF -> CDN -> CLB
    |
TKE Ingress
    |
├── Web 应用
├── API 服务
├── AI Worker
└── Report Service
    |
    +--> TencentDB PostgreSQL
    +--> TencentDB Redis
    +--> TDMQ/CMQ
    +--> COS
    +--> KMS / Secrets Manager
    +--> CLS / APM / 云监控
```

生产原则：

- 当前推荐商用形态为 `CVM + COS + 云端公网域名入口`；当租户数、并发和运维复杂度增长后，再演进到 `CLB + 多 CVM` 或 `TCR + TKE`。
- 服务启动、发布、扩容、回滚全部发生在云端，不依赖开发机、办公室网络或 SSH 常驻终端。
- 文件和报告以 COS 为唯一生产持久化存储，不以应用本地磁盘为生产真源。
- Web、API、下载和小程序请求统一收敛到公网云接入链路，减少单机出口和本地网络波动对服务可用性的影响。

## 4. 当事人端案件详情页（重点设计）

### 4.1 页面入口

- 当事人登录后默认进入案件列表页（无多 Tab 干扰）。
- 点击案件卡片进入案件详情页。

### 4.2 页面布局（自上而下）

```text
┌─────────────────────────────────────────┐
│ ← 返回          案件详情                 │
├─────────────────────────────────────────┤
│ 【案件概览卡片】                          │
│   案号 / 名称 / 律师 / 创建时间 / 状态     │
├─────────────────────────────────────────┤
│ 【时间流】                                │
│   展示解析、上传、补充、律师操作等事件     │
├─────────────────────────────────────────┤
│ 【AI 分析摘要】                           │
│   完成后展示摘要；未完成展示进度条与预估    │
├─────────────────────────────────────────┤
│ 【证据材料列表】                          │
│   文件名 / 类型 / 解析状态 / 描述 / 预览    │
├─────────────────────────────────────────┤
│ 【操作按钮区】                            │
│   [下载完整报告(PDF)] [补充材料]           │
└─────────────────────────────────────────┘
```

### 4.3 模块详细规则

#### 1) 案件概览卡片

- 字段：案号、案件名称、负责律师、创建时间、解析状态。
- 解析状态显示规则：
  - `解析中 xx%`：显示实时百分比与阶段名。
  - `解析完成`：绿色徽章。
  - `解析失败`：红色提示并引导稍后重试或联系律师。

#### 2) 时间流（关键模块）

- 与 Web 端使用同一张时间流表，仅展示当事人可见记录。
- 操作类型至少包括：
  - 系统解析开始/完成
  - 用户上传文件/添加描述
  - 律师上传新证据
  - 律师修改案件信息
  - AI 缺失证据提示
- 每条记录字段：时间、操作者（您/律师/系统）、操作内容。
- 解析过程建议拆分阶段：上传完成 -> OCR -> 切片 -> 向量化 -> 总结生成。

#### 3) AI 分析摘要

- 仅在解析完成后展示，内容与 Web 端客观摘要一致，不给法律建议。
- 内容结构：
  - 事件经过（按时间线组织）
  - 缺失证据提示（基于证据清单对比）
- 未完成时展示进度条与预计剩余时间（按文件数与历史耗时估算）。

#### 4) 证据材料列表

- 默认仅展示当事人自己上传的文件。
- 律师上传文件可选策略：
  - 方案 A：完全隐藏。
  - 方案 B：仅显示文件名，不可下载。
- 每条显示：文件图标、文件名、解析状态、描述、预览入口。

#### 5) 操作按钮

- 下载完整报告（PDF）：
  - 调用后端生成并返回临时下载链接。
  - 报告包含案件基本信息、AI 摘要、证据清单、时间流。
  - 自动隐藏内部管理字段。
- 补充材料：
  - 跳转上传页，支持多文件和逐文件描述。
  - 上传成功后自动触发重解析（带防抖限流）。

| 功能 | Web 端（律师） | 小程序端（当事人） | 数据源 |

- 当事人端查询条件：`visible_to in ('client','both')`。

## 6. 关键技术实现

### 6.1 PDF 报告生成

- 接口：`GET /api/v1/cases/{caseId}/report`
- 流程：后端生成 PDF -> 写入 COS -> 返回签名 URL 或 CDN 授权 URL。
- 模板：按 `role=lawyer|client` 动态选择字段。
- 生成方案：`Node + Puppeteer(HTML)`（独立服务部署）。
- 小程序下载：`wx.downloadFile` + `wx.openDocument`。

### 6.2 解析进度实时推送

- 后端在任务阶段更新 `analysis_status` 与 `analysis_progress`。
- 推送方式：
  - 主通道：WebSocket 实时推送。
  - 备通道：轮询拉取最新状态与时间流。

### 6.3 时间流性能优化

- 案件详情页初次加载拉取第一页，按时间倒序。
- 长列表支持分页滚动加载。
- 关键事件（解析完成、失败、缺失证据）高亮展示。

### 6.4 补充材料后的重解析策略

- 状态切换为“待重新解析”。
- 防抖规则：短时间多次补充仅触发一次任务。
- 保留历史解析版本，时间流记录“重新解析”事件。

### 6.5 成本控制与熔断（MVP 必做）

- 生产异步执行采用云消息队列 + 独立 worker，开发与过渡环境可保留 DB 队列（`ai_tasks` 状态机 + worker）作为兼容模式。
- 任务失败进入重试流程，超过阈值进入死信状态（`status=dead`）。
- 队列驱动可配置（`db|cmq|tdmq`），本地默认 `db`，生产目标为 `cmq/tdmq`。
- 新增租户月度预算上限：`ai_monthly_budget_limit`（元）。
- 新增单案件预算上限：`ai_case_budget_limit`（元），超限即阻断该案件继续解析。
- 熔断策略：
  - 先降级到低成本模型（可配置），并记录 `analysis_model_downgraded` 时间流；
  - 若降级后仍超预算，停用该租户或案件的自动解析，并记录 `analysis_budget_circuit_open`；
  - 人工手动触发时返回明确错误码与可读提示。

## 8. AI 解析主流程

1. 前端向后端申请上传凭证，直传文件到 COS。
2. 后端确认对象元数据并记录 `case_files`，状态为 `pending`。
3. 写入云消息队列或兼容队列（`status=queued`），由云端 worker 异步消费。
4. 消费者执行：
   - 从 COS 拉取文件并进行类型识别
   - OCR / 文档解析 / ASR
   - 文本聚合（含用户描述）
   - 文本切片与向量化
   - 大模型生成摘要与缺失证据
5. 写入 `ai_analysis`，更新案件解析状态。
6. 失败任务按重试策略处理，超过阈值转入死信状态（`status=dead`）。
7. 写入 `case_flows` 并推送前端刷新。

约束：

- 输出必须“客观总结，不提供法律建议”。
- 支持版本号，保留重解析历史。

## 11. 部署与运维

- 环境：开发/测试/生产分离。
- 当前推荐商用运行面：`CVM + Docker Compose`。
- 当前推荐商用公网入口：`DNSPod/域名 + SSL + Nginx`，下载量或跨地域访问上来后再补 `CDN`。
- 当前推荐商用存储面：`COS` 作为案件材料、报告、导出文件唯一生产真源。
- 当前推荐数据面：预算有限时使用同机 `PostgreSQL + Redis`；更稳妥时切换到 `TencentDB PostgreSQL + TencentDB Redis`。
- 当前推荐计费策略：`CVM 按流量计费`，并通过 `COS/COS + CDN` 卸载文件流量，避免附件下载长期压在 CVM 公网。
- 演进运行面：`CLB + 多 CVM` 或 `TCR + TKE + WAF + CDN`。
- 演进数据面：`TencentDB PostgreSQL + TencentDB Redis + TDMQ/CMQ + COS`。
- CI/CD：首阶段允许 `CVM` 上 `Docker Compose pull/up` 或镜像拉起；后续演进为 `测试 -> 构建镜像 -> 推送 TCR -> 发布到多 CVM / TKE`。
- 监控：APM + CLS + 云监控 + 队列堆积与对象存储错误告警。
- 密钥：首阶段至少完成主机环境变量收口、访问密钥最小权限化；后续把生产敏感配置迁移到 `KMS / Secrets Manager`。
- 备份：数据库全量与 PITR，`CVM` 快照与自动备份，COS 开启版本控制或生命周期策略。

## 12. 迭代策略与非目标

### 12.1 MVP 优先级

- 优先打通主链路：上传文件 -> AI 解析 -> 当事人查看 -> 下载报告。
- 保证两端数据一致与权限隔离。
- 完成案件详情页关键体验：可见进度、时间流、摘要、补充材料。

### 12.2 云端实施顺序

- 第一波：订阅 `VPC/CVM/COS/域名/SSL`，用 `Docker Compose` 将现有服务启动到 `CVM`，保证业务先在云端独立运行。
- 第二波：文件与报告迁移到 `COS`，前端改为直传/签名下载，服务器不再作为大文件长期中转。
- 第三波：按预算和稳定性需要，补齐 `CDN`、托管 `PostgreSQL/Redis`、第二台 `CVM` 或 `CLB`。
- 第四波：当租户数、下载量、异步任务规模增长后，继续演进到 `TCR/TKE` 与 `TDMQ/CMQ`，补齐发布、回滚和弹性扩缩。

### 12.3 当前非目标（暂不做）

- 自定义流程编排引擎。
- 高复杂度计费系统（先做配额扣减）。
- 端侧离线解析与本地模型推理。

## 13. 验收标准（针对本蓝图）

- 当事人端案件详情与 Web 端同源数据一致。
- 时间流可见性控制准确，无越权数据泄露。

## 16. 角色体系与准入流程（V3）

### 16.1 角色定义

| 角色 | 标识 | 主要职责 |
| --- | --- | --- |
| 超级管理员 | `super_admin` | 平台级租户、计费、全局配置与风控 |
| 机构管理员（机构创建者） | `tenant_admin` | 律所管理、律师审批、业务配置 |
| 机构律师 | `org_lawyer`（归一到 `lawyer`） | 办案与当事人协同，不含机构管理 |
| 单独律师 | `solo_lawyer`（归一到 `lawyer`） | 个人租户内办案与 AI 使用 |
| 当事人 | `client` | 小程序上传材料、查看进度、下载报告 |

### 16.2 注册与登录规则

- 所有 Web 注册均需“手机号 + 短信验证码 + 密码”。
- 密码规则：8-128 位，必须包含字母和数字，不允许空格。
- 手机号规则：11 位中国大陆手机号，前后端双重校验。
- 机构律师不允许开放注册：
  - 必须由机构管理员发邀请；
  - 通过邀请链接完成注册；
  - 注册后状态为待审批，审批通过后可登录。
- 单独律师可直接注册，不要求填写机构邀请码。

### 16.3 小程序微信直登演进规则

- Web 端继续使用“手机号 + 密码 + 短信校验”，本次变更只改小程序认证方式。
- 小程序端目标改为“微信号直登”：
  - 第一步：调用 `wx.login` 获取登录 `code`；
  - 第二步：后端通过 `jscode2session` 换取 `openid / unionid`；
  - 第三步：用户点击 `getPhoneNumber` 授权手机号；
  - 第四步：后端以“微信身份 + 微信手机号 + invite token / tenant_code”完成账号绑定或创建。
- 绑定规则：
  - 律师 / 机构管理员：只允许绑定已存在账号，不开放通过微信自由创建管理账号；
  - 机构律师仍必须先邀请注册并审批，审批后才允许在小程序完成微信绑定；
  - 当事人拿到案件邀请 token 后，可自动创建或绑定 `client` 账号并关联案件。
- 邀请路径规则：
  - 案件邀请进入小程序时，路径必须显式带 `scene=client-case`；
  - 机构律师邀请注册链接继续沿用普通 `token` 参数，避免与案件邀请 token 混淆；
  - 小程序登录页需根据 `scene` 区分“案件绑定”与“机构律师邀请注册”两条入口。
- 多租户冲突规则：
  - 同一手机号命中多个租户时，不自动猜测租户；
  - 必须通过 `tenant_code` 或案件邀请 token 明确落点；
  - 若仍无法唯一确定，返回冲突提示，不直接登录。
- 登出规则：
  - Web 与小程序都必须提供显式“退出登录”入口；
  - 客户端退出时清理本地 access token、refresh token、用户缓存；
  - 服务端撤销当前 refresh session，access token 走短时自然过期。

- 小程序需与 Web 保持同构能力：机构管理员 5 菜单、机构律师/单独律师 4 菜单。
- 小程序额外提供“当事人端”独有入口与业务链路（Web 不提供该角色界面）。
- 小程序登录方式升级为“微信直登 + 微信手机号授权”，不再把手机号密码表单作为长期主入口。
- 当事人必须通过律师邀请链接进入并绑定案件；律师 / 管理员则通过已存在账号与微信身份自动绑定。

### 18.2 律师/管理员同构主链路（必须与 Web 对齐）

1. 概览：用户信息卡、律所统计卡、较上次登录增量卡。
2. 案件管理：列表筛选/排序/搜索、新建案件（`legal_type` 必填）、截止时间 30/7 预警。
3. 案件详情：概览、全量时间流、证据列表、AI 进度与结果、报告下载。
4. 当事人管理：列表、详情、编辑、案件回跳。
5. 律师管理（机构管理员/超管）：邀请、审批、启停、详情行为记录。
6. 分析管理：用量可视化、提示词管理、Provider/API 设置。

### 18.3 当事人附加主链路（小程序独有）

1. 邀请进入 / 打开小程序 -> `wx.login` -> 微信手机号授权 -> 自动绑定 / 激活账号；
2. 进入案件列表（单案直达详情、多案进列表）；
3. 案件详情查看：概览、可见时间流、解析进度、解析结果、证据；
4. 补充材料：多文件选择 + 逐文件描述 + 确认提交；
5. 自动触发解析，实时显示进度；
6. 下载最新 PDF 报告（仅最新版本）。

### 18.4 权限边界

- 律师/管理员：权限与 Web 同角色一致（同一后端鉴权规则）。
- 当事人：只能看本人案件，且只能看 `visible_to in ('client','both')` 的时间流。
- 当事人报告：仅可下载最新版本；律师可见历史版本。
- 律师上传材料对当事人默认“仅显示名称不可下载”（按策略可切换）。
- 小程序工作台与当事人端都必须提供显式退出登录入口。

## 23. 小程序 V4 体验重构方案（Apple 风格）

> 目标：在不改变“Web 与小程序共享同一后端、同一权限模型、同一业务主链路”的前提下，
> 将小程序重构为更清晰的双工作台：律师/机构管理员工作台与当事人服务台。  
> 要求：律师与机构管理员在小程序中的功能必须与 Web 端保持一致；小程序仅额外增加当事人端链路。

### 23.1 设计原则

- 双端同构：律师/机构管理员看到的模块、字段、核心操作、权限边界必须与 Web 端一致，仅交互形态可针对小屏优化。
- 单页单主任务：每个页面只保留一个主要操作，避免同屏堆叠多个高权重入口。
- 渐进披露：先列表、后详情、再编辑；低频设置收敛到二级页或操作面板，不在首页混放。
- 稳定导航：导航结构固定，角色切换后菜单数量和顺序保持可预期，不混入一次性入口。
- 明确反馈：上传、解析、失败、预算熔断、权限不足都必须有明确状态和可理解文案。
- Apple 风格取向：强调留白、分组、层级、克制的色彩和一致的动效反馈，不做复杂装饰性 UI。

### 23.2 信息架构重构

小程序拆分为两个工作区：

1. 律师/机构管理员工作台
2. 当事人服务台

其中：

- 律师/机构管理员工作台必须与 Web 保持同一业务骨架。
- 当事人服务台只保留“受邀绑定 -> 补充材料 -> 查看进度 -> 查看解析结果 -> 下载最新 PDF”链路。
- AI Provider、预算、提示词等管理能力仍走统一后端，不新增小程序专属后端。

### 23.3 角色与导航规则

#### 1) 律师/机构管理员导航

采用底部 TabBar + 页面内分组的方式，保证导航稳定且符合小程序交互习惯。

- `tenant_admin`：`概览` / `案件` / `当事人` / `律师` / `分析`
- `lawyer`：`概览` / `案件` / `当事人` / `分析`

规则：

- 模块名称、顺序、权限判定与 Web 完全一致。
- 登录成功后默认进入 `概览`，不再混入“临时入口页”。
- 所有详情页均从对应模块进入，返回路径稳定，不出现跨模块跳转迷路问题。

#### 2) 当事人导航

当事人不使用律师工作台导航，只保留案件服务链路。

- 单案件：登录后直接进入案件详情。
- 多案件：先进入案件列表，再进入案件详情。
- 详情页内固定保留三个核心区域：案件概览、材料/时间流、AI 结果与最新报告。

规则：

- 当事人不显示律师管理、分析管理、Provider 设置等入口。
- 当事人的主操作始终是“补充材料”或“查看最新进度”。
- 当事人只能看到最新报告，不展示历史报告版本。

### 23.4 页面层级与职责重构

#### A. 律师/机构管理员工作台

律师与机构管理员首页结构必须统一，区别只在是否显示“律师管理”模块。

1. `概览`
   - 当前登录人信息卡
   - 律所/个人执业信息卡
   - 核心指标：案件总数、当事人总数、待审批律师数、近 30 天 AI 调用量
   - 动态提醒：较上次登录新增案件、待处理材料、临近截止案件、待审批律师
2. `案件`
   - 案件列表、搜索、筛选、排序
   - 新建案件入口
   - 截止时间预警与状态色
   - 进入案件详情后展示概览、材料、时间流、解析记录、报告区
3. `当事人`
   - 当事人列表
   - 当事人详情
   - 当事人资料更新
   - 从当事人详情回跳到案件详情
4. `律师`（仅 `tenant_admin`）
   - 邀请机构律师
   - 审批注册申请
   - 成员启用/停用
   - 律师详情与权限说明
5. `分析`
   - AI 使用量可视化
   - 三类提示词编辑
   - Provider/API 设置
   - 预算熔断与降级模型配置入口

补充规则：

- 不再设置“杂项功能页”或“功能大杂烩首页”。
- 列表页负责发现，详情页负责理解，编辑页负责修改。
- 删除、停用、重新解析等高风险操作统一放在操作菜单中，先确认再执行。

#### B. 当事人服务台

当事人端只围绕案件协作展开，不承担管理职责。

1. `案件列表/案件直达`
   - 单案件直接进入详情
   - 多案件显示案件卡片列表
   - 卡片显示案件名称、案号、状态、最新更新时间、负责律师
2. `案件详情`
   - 案件概览
   - 时间流
   - 已上传材料列表
   - AI 解析摘要
   - 最新 PDF 报告下载入口
3. `补充材料`
   - 选择文件
   - 可选填写补充说明
   - 批量确认上传
   - 上传后自动触发解析，并显示防抖说明与进度状态

补充规则：

- 当事人详情页即主页面，不再拆成过多子页面。
- 当事人所有可执行动作统一收敛到一个主按钮：`补充材料`。
- 上传完成后自动回到案件详情，优先展示最新进度而非再次停留在表单页。

### 23.5 权限呈现策略

权限采用“后端强校验 + 前端显式隐藏 + 越权提示”的组合策略。

#### `tenant_admin`

- 可查看本租户全量案件、当事人、律师、AI 使用量。
- 可管理机构律师邀请、审批、启停。
- 可编辑租户级提示词、预算参数、Provider 配置（若系统未锁定云侧配置）。

#### `lawyer`

- 可查看和操作自己有权限的案件与当事人。
- 可发起文件上传、AI 解析、查看历史报告。
- 可进入分析管理查看使用量与当前策略。
- 默认不可修改租户级 Provider/API 设置，避免越权。

#### `client`

- 仅可访问自己被关联的案件。
- 可补充材料、查看解析进度、查看 AI 解析摘要、下载最新 PDF。
- 不可查看历史报告，不可触发租户级配置修改，不可访问其他角色页面。

### 23.6 Apple 风格视觉系统规范

#### 视觉基线

- 页面背景：`#F5F5F7`
- 卡片背景：`#FFFFFF`
- 主色：`#0071E3`
- 成功：`#34C759`
- 警告：`#FF9F0A`
- 危险：`#FF3B30`
- 主文字：`#1D1D1F`
- 次级文字：`#6E6E73`

#### 组件与布局

- 使用大标题 + 分组卡片，而不是长段落和密集表格。
- 统一 16/20 px 间距体系，卡片圆角与阴影保持克制。
- 列表项尽量采用“标题 + 辅助信息 + 右箭头”结构，降低理解成本。
- 主要操作按钮固定在底部安全区或卡片主操作位，不在同页放多个主按钮。
- 筛选、排序、更多操作优先使用分段控件、Sheet、Popover，不使用复杂弹窗堆叠。

### 23.7 与现有蓝图和后端契约的关系

本次 V4 重构只调整小程序信息架构与交互，不改变以下底层约束：

- 继续复用现有 API，与 Web 端共享同一数据模型。
- 继续沿用租户隔离、用户隔离、角色只读/可写边界。
- 律师/机构管理员在小程序的能力必须与 Web 保持同构。
- 当事人端仍是小程序独有链路，但数据来源与案件主链路不分叉。
- 报告可见性继续遵守：律师可见历史、当事人仅见最新。

### 23.8 推荐实施顺序

1. 先重构导航与路由守卫，建立“律师工作台 / 当事人服务台”双工作区。
2. 再重构概览、案件、当事人、律师、分析五个核心模块的页面层级。
3. 单独收口当事人主链路：登录/绑定 -> 案件 -> 补材 -> 解析 -> 最新报告。
4. 最后统一视觉系统、状态反馈、空态/错误态、登录与 token 续期体验。
5. 完成 HBuilderX、微信开发者工具、真机的三端验收后，才进入云部署联调。

## 24. 安全加固波次 S1（2026-03-24）

### 24.1 本轮已落地

- 环境安全基线：生产/准生产环境禁止默认 `SECRET_KEY`、禁止 `localhost` CORS、禁止 `AI_MOCK_MODE=True`。
- 鉴权收敛：Access Token 在生产环境要求携带 `sid`，并在鉴权阶段回查 `auth_sessions`，登出时默认撤销当前会话。
- 小程序来源校验统一：`routes_auth` 与 `dependencies/auth` 统一使用同一套 `X-Client-Platform + X-Client-Source` 判定逻辑。
- 租户上下文修正：租户上下文写入 `Session.info`，并在每个新事务开始时自动重放，避免 `commit` 后 RLS 上下文丢失。
- 角色别名查询收敛：`lawyer / org_lawyer / solo_lawyer` 统一通过查询 helper 展开，避免组织律师被漏匹配。
- 文件访问加固：文件列表不再暴露固定 `/download` 入口，改为 `access-link`；本地存储下载令牌改为一次性消费，防止重复重放。
- AI 任务运行基线：`AI_DB_QUEUE_EAGER` 默认关闭；显式开启时默认改为后台执行，仅在 `AI_DB_QUEUE_EAGER_BLOCKING=True` 时允许阻塞式内联执行。
