## 1. 里程碑划分

### M1：基础能力与数据底座（P0）

- [ ] 完成统一鉴权：JWT 登录、刷新、登出。
- [ ] 完成租户隔离中间件：所有查询自动注入 `tenant_id` 条件。
- [ ] 完成角色权限基线：律师、机构管理员、当事人权限边界。
- [ ] 建立核心数据表：`users`、`cases`、`case_files`、`case_flows`、`ai_analysis`、`tenants`。
- [ ] 建立核心索引与唯一约束（案号唯一、手机号唯一、案件时间流索引）。

验收标准：

- [ ] 越权访问用例全部拦截（至少 10 个反向用例）。
- [ ] 数据库迁移可一键执行，测试环境可回滚。

### M2：Web 端案件主流程（P0）

- [ ] 案件列表页：分页、筛选、搜索、状态显示。
- [ ] 案件详情页：概览卡、全量时间流、证据列表。
- [ ] 上传证据：后端签名 + 前端直传 COS + 回调入库。
- [ ] 手动触发解析按钮与任务状态反馈。
- [ ] 邀请当事人链接生成（带 caseId/加密参数）。

验收标准：

- [ ] 律师可在 Web 完整查看案件全链路记录。
- [ ] 上传文件后 3 秒内可在详情页看到文件记录（不含 AI 完成时间）。

### M3：当事人端小程序 MVP（P0）

- [ ] 邀请落地页与手机号登录，自动绑定案件。
- [ ] 案件列表页（无多 Tab）：卡片展示案号、名称、状态、更新时间。
- [ ] 案件详情页（重点）：概览卡、时间流、AI 摘要、证据列表、操作按钮。
- [ ] 补充材料页：多文件选择、逐文件描述、提交确认。
- [ ] 文件预览能力（图片/PDF/常见文档）。

验收标准：

- [ ] 当事人端只可见本人关联案件。
- [ ] 当事人端时间流仅显示 `visible_to in ('client','both')` 数据。
- [ ] 当事人端证据列表只显示本人上传（或按配置仅显示律师文件名）。

### M4：AI 异步解析与实时进度（P0）

- [ ] DB 队列模型落地：`ai_tasks` 入队 + 独立 worker 消费（不依赖云 MQ）。
- [ ] 队列状态流转：`queued -> processing -> completed|retrying|dead`。
- [ ] 队列策略：失败重试、退避、幂等控制、死信阈值。
- [ ] 解析流水线：OCR/文档解析/ASR -> 文本聚合 -> 向量化 -> 摘要生成。
- [ ] 状态字段落库：`analysis_status`、`analysis_progress` 分阶段更新。
- [ ] 时间流自动写入：解析开始、阶段推进、解析完成、解析失败。
- [ ] 成本熔断：月预算上限、单案件预算上限、超额降级模型/停用自动解析。
- [ ] 队列驱动可配置：`QUEUE_DRIVER=db|cmq`（MVP 默认 `db`）。
- [ ] 实时推送：WebSocket 主通道 + 轮询兜底。

验收标准：

- [ ] 单案件多文件解析过程中，进度与时间流可持续更新。
- [ ] 解析失败可观测（错误码+错误信息+重试入口）。

### M5：PDF 报告与交付闭环（P0）

- [ ] 报告接口：`GET /api/v1/cases/{caseId}/report`。
- [ ] 角色模板：律师版、当事人版字段差异化渲染。
- [ ] COS 存储与 24 小时临时下载 URL。
- [ ] 小程序端下载与文档打开流程。

验收标准：

- [ ] 当事人报告不包含内部管理字段。
- [ ] 报告内容与数据库实时数据一致（同一时刻生成）。

### M6：上线与质量门禁（P0）

- [ ] API 自动化测试（核心接口覆盖率 >= 80%）。
- [ ] 关键链路 E2E：Web 创建案件 -> 邀请当事人 -> 小程序上传 -> AI 完成 -> 下载报告。
- [ ] 日志与监控：错误率、任务积压、接口耗时、推送失败率。
- [ ] 生产发布脚本与回滚预案。

验收标准：

- [ ] 发布后核心链路可在生产环境复现通过。
- [ ] 具备故障可观测与可回滚能力。

## 2. 任务分解（按模块）

## 2.1 后端（P0）

- [ ] `Auth`：登录、刷新 Token、权限注解、角色校验。
- [ ] `Case`：案件 CRUD、状态流转、列表筛选。
- [ ] `File`：COS 上传签名、回调入库、预览链接。
- [ ] `Flow`：统一写入 `case_flows`，支持 `visible_to`。
- [ ] `Analysis`：异步任务创建、状态更新、结果写入 `ai_analysis`。
- [ ] `Report`：按角色生成 PDF 并返回下载地址。
- [ ] `Realtime`：WebSocket 推送与轮询接口。

## 2.2 Web 前端（P0）

- [ ] 登录页与角色路由守卫。
- [ ] 案件列表页（筛选、排序、检索、分页）。
- [ ] 案件详情页（概览、时间流、证据、AI 摘要）。
- [ ] 上传与解析触发交互（含进度态、异常态）。
- [ ] 邀请当事人弹窗与链接复制。

## 2.3 小程序前端（P0）

- [ ] 邀请链接落地与账号绑定流程。
- [ ] 案件列表页与案件详情页。
- [ ] 时间流组件（分页加载、关键事件高亮）。
- [ ] AI 摘要组件（完成态/进行态）。
- [ ] 证据列表与文件预览。
- [ ] 补充材料上传页（多文件+描述+确认提交）。
- [ ] 报告下载/打开流程。

## 2.4 数据与运维（P0）

- [ ] 数据库迁移脚本与初始化脚本。
- [ ] Redis 键设计与过期策略。
- [ ] DB 队列 worker 策略：拉取频率、重试退避、死信阈值、并发配置。
- [ ] 环境变量模板（dev/test/prod）。
- [ ] CLS/APM/告警配置。

## 3. 非功能任务清单

- [ ] 安全：上传文件类型白名单、大小限制、恶意文件扫描。
- [ ] 安全：接口限流（登录、短信、报告下载）。
- [ ] 性能：案件详情与时间流分页查询优化。
- [ ] 可靠性：解析任务幂等键与重复消费保护。
- [ ] 成本：租户月预算上限、单案件预算上限、超额熔断与模型降级策略。
- [ ] 可观测：全链路 `request_id`、任务 `trace_id`。
- [ ] 合规：日志脱敏（手机号、证件号、路径签名）。

- [ ] 用例 5：当事人下载 PDF 报告，字段符合当事人角色模板。
- [ ] 用例 6：律师重解析后，保留历史版本并新增时间流记录。
- [ ] 用例 7：当事人访问非本人案件，返回 403。

## 6. MVP 完成定义（Definition of Done）

- [ ] 主链路“上传 -> 解析 -> 查看 -> 下载”端到端可用。
- [ ] Web/小程序查询同源，权限过滤准确无越权。
- [ ] 时间流和解析进度可实时感知。
- [ ] 成本熔断策略可验证生效（降级/停用自动解析 + 可追踪日志/时间流）。
- [ ] 关键接口、关键页面、关键任务均有测试覆盖。
- [ ] 生产环境具备监控、告警、备份、回滚能力。

#### 7.5.3 V5-AUTH 验收口径

- [ ] 已绑定微信身份的律师 / 管理员 / 当事人，再次进入小程序时无需手输手机号密码。
- [ ] 当事人带邀请 token 首次登录时，可自动完成账号绑定并进入单案直达 / 多案列表。
- [ ] 同手机号命中多个租户时，系统必须中断自动登录并要求 `tenant_code` 或邀请 token。
- [ ] 待审批律师即使微信身份已绑定，仍不能越过审批直接进入工作台。
- [ ] Web / 小程序退出登录后，当前 refresh session 失效，重新打开页面不能静默继续访问。

#### V4 验收要点

- [ ] 律师/机构管理员在小程序中的模块、顺序、权限与 Web 保持一致。
- [ ] 当事人只保留案件服务链路，且单案件直达、多案件列表逻辑稳定。
- [ ] 每个页面只有一个主操作，返回路径清晰，不出现角色混页和导航迷路。
- [ ] 上传、解析、失败、超额停用、权限不足等状态反馈一致且可理解。
- [ ] 视觉风格统一，卡片、状态色、按钮、空态、错误态遵守同一设计系统。
- [ ] 三端验收通过：HBuilderX、微信开发者工具、真机表现一致。
