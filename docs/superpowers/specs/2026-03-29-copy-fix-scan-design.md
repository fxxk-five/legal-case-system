# 文案修复与主路径可用性治理设计

> 本设计文档聚焦 P0 文案修复与首登改密缺陷修复，并完成 P1 的“全仓污染扫描 + 静态扫描脚本”。

**目标**
- 让用户主路径的可见文字不再乱码，首登改密链路可用。
- 对全仓用户可见文案污染进行一次性收口，并提供静态扫描脚本防回归。

**范围**
- P0：Web 主壳、导航、通知、退出登录、强制改密、登录失败提示、案件详情操作提示。
- P0：后端直接返回前端的错误文案（微信登录票据、案件邀请令牌、ASR 上传格式校验）。
- P0：首登改密页“手机号展示位坏模板文本”缺陷修复及回归验证。
- P1：对 `web-frontend/src`、`mini-program`、`backend/app` 进行文案污染扫描并修复。
- P1：提供静态扫描脚本（本地/CI 可用），检测常见乱码特征串并失败退出。

**非目标**
- 不在本轮完成文案单一源收口（P1 后续）。
- 不在本轮补主路径 smoke（P1 后续）。
- 不做功能性重构与视觉重构，仅修正文案与明确缺陷。

---

## 现状与问题
- 多处 UI 文案出现乱码或模板残留，覆盖主壳、导航与强制改密页面。
- 后端部分错误消息对用户不可读。
- 缺少“乱码回归”的静态扫描手段。

## 方案概述
1. **P0 文案修复**：对指定页面/模块的用户可见文本替换为正式简体中文。
2. **首登改密修复**：修复手机号展示位坏模板文本，确保提示与校验信息可读。
3. **后端错误文案修复**：对微信登录票据、案件邀请令牌、ASR 上传格式校验的消息进行可读化。
4. **全仓扫描与脚本**：对指定目录进行乱码特征串扫描，修复后补脚本用于本地/CI 阻断。

## 设计细节

### P0 文案修复
- **Web 主壳/导航/通知/退出**：集中修复 `DashboardLayout.vue` 与 `accessControl.js` 中的可见文案。
- **强制改密页**：修复 `ForceResetPasswordPage.vue` 中标题、说明、占位符、提交态提示、成功/失败提示等。
- **后端错误文案**：修复 `token_service.py` 的 `message` / `detail`。
- **登录失败提示与案件详情操作提示**：通过搜索定位具体文件，统一为正式简体中文。

### P1 扫描与脚本
- **扫描范围**：
  - `web-frontend/src`
  - `mini-program`
  - `backend/app`
- **检测策略**：
  - 只扫描文本/源码文件，过滤常见二进制/缓存（例如 `.png` `.jpg` `.pyc` `.zip` `.pdf`）。
  - 检测常见乱码特征串：`锟`、`?`、`?`、`?`、`a` 等。
- **脚本输出**：
  - 打印命中文件与行号；若存在命中则以非零退出码结束，用于 CI 阻断。

## 数据与流程
- 仅涉及静态文案替换；不会改变业务流程或数据结构。
- 首登改密的流程保持不变，仅修正文案与可见缺陷。

## 错误处理
- 后端错误消息使用简洁明确的用户语言，保持 `message` 与 `detail` 一致。
- 前端失败提示统一为可读短句，避免技术栈信息泄露。

## 测试与回归
- **P0 必须回归**：
  - 未登录跳转到登录页
  - 必须改密用户被正确拦截
  - 改密成功后重建会话并跳转
  - 改密失败提示清晰可读
  - 登录失败提示可读
- **P1 扫描验证**：
  - 运行静态扫描脚本，确保无命中项

## 风险与缓解
- **风险**：误将二进制文件识别为乱码文本。
- **缓解**：脚本严格排除二进制扩展名与不可读文件。

## 交付清单

- 修复后的后端错误文案
- 乱码静态扫描脚本
- 相关文档更新（如需）
