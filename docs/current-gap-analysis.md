# 现状差距清单

本文档用于回答一个核心问题：

当前仓库，距离你最初定义的完整系统，还差什么。

## 一、总体判断

当前仓库不是“从零开始”，也不是“即将上线”。

更准确的判断是：

- 已完成一个能跑通主流程的本地演示版
- 已部分覆盖阶段一、阶段二、阶段三
- 阶段四、阶段五、阶段六大部分还未真正落地

## 二、按阶段看差距

## 阶段一：基础架构与多租户

### 已有

- FastAPI 项目结构
- SQLAlchemy 模型
- Alembic 迁移
- JWT 登录
- 基础租户、用户、案件、文件模型

### 缺失

- `POST /tenants/personal`
- `POST /tenants/organization`
- `POST /tenants/join`
- 管理员审批加入机构流程
- 完整的 RLS 策略与数据库上下文注入
- 去除默认租户写死逻辑
- 更完整的租户生命周期管理

### 结论

阶段一只完成了“骨架版”，没有完成“正式可扩展版”。

## 阶段二：核心业务逻辑（律师 Web 端）

### 已有

- 登录
- 案件列表
- 案件详情
- 文件查看
- 律师管理
- 机构设置
- 通知和统计的基础接口

### 缺失

- 案件看板拖拽
- 更完整的案件进度管理
- AI 正式入口与结果展示规范
- 订阅管理与续费流程
- 实时通知或稳定轮询策略
- 更完善的空态、异常态、加载态

### 结论

阶段二完成度中等，能演示，但离高频使用还有距离。

## 阶段三：小程序端

### 已有

- uni-app 项目骨架
- mock 微信登录
- 手机号绑定
- 律师案件列表
- 新建案件
- 当事人进入案件
- 文件上传
- 文件预览与下载基础逻辑

### 缺失

- 个体律师注册
- 律师加入机构与审批状态查询
- 正式微信登录
- 二维码保存与分享增强
- 个人中心
- 更完整的通知中心
- 更稳定的真机联调方案

### 结论

阶段三目前更像“演示闭环版”，不是“完整小程序产品版”。

## 阶段四：AI 与计费

### 已有

- 少量 AI 占位思路

### 缺失

- 真实大模型调用
- Prompt 模板管理
- Redis / Celery 异步任务
- AI 使用记录
- 个人余额扣费
- 机构套餐配额扣减
- 支付回调与幂等处理

### 结论

阶段四基本未开始。

## 阶段五：平台超级管理员后台

### 已有

- 无正式平台后台

### 缺失

- 超级管理员认证
- 全租户管理
- 全用户管理
- 套餐配置
- 系统参数配置
- 平台收入和增长报表

### 结论

阶段五基本未开始。

## 阶段六：测试、部署与上线

### 已有

- Dockerfile
- docker-compose
- Nginx 基础配置
- 本地 Docker 演示
- 腾讯云部署说明文档

### 缺失

- 自动化测试体系
- 预发布环境
- 安全审计
- 监控和告警
- 备份恢复演练
- 正式对象存储、正式 HTTPS、正式域名联调
- 微信审核与上线材料闭环

### 结论

阶段六只完成了“部署准备文档”和“本地容器演示”，没有完成正式上线闭环。

## 三、代码层面最值得优先重构的问题

1. 认证与租户逻辑仍存在默认租户思维
2. 租户创建、加入、审批没有形成完整业务链路
3. 小程序很多流程仍是为演示方便做的折中实现
4. 文件系统仍偏向本地存储思路，没有切到正式对象存储策略
5. AI、支付、平台运营能力没有进入主工程

## 四、后续优先级建议

### 第一优先级

- 多租户
- 审批流
- RLS
- 正式权限模型

### 第二优先级

- Web 案件看板
- 小程序完整角色流程
- 文件权限与对象存储策略

### 第三优先级

- AI
- 支付
- 套餐
- 超级管理员后台

### 第四优先级

- 自动化测试
- 监控
- 安全
- 上线
