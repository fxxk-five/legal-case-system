# 项目结构与角色流程

本文档帮助你从“工程结构”和“用户使用流程”两个角度看当前项目，并快速发现问题。

## 一、项目结构

## 1. `backend/`

这是后端主服务。

### 主要目录

- `app/api/`：接口路由
- `app/models/`：数据库模型
- `app/schemas/`：请求和响应数据结构
- `app/services/`：业务服务
- `app/dependencies/`：认证和权限依赖
- `app/scripts/`：初始化、演示数据、通知生成脚本
- `alembic/`：数据库迁移

### 当前问题

- 租户相关接口还不完整
- AI、支付、平台管理模块还没真正形成独立子域

## 2. `web-frontend/`

这是律师 Web 端和部分机构管理员功能所在位置。

### 主要内容

- 登录页
- 案件列表
- 案件详情
- 管理后台布局
- 律师管理
- 机构设置

### 当前问题

- 还没有真正的案件看板
- 管理能力偏演示
- 缺少平台超级管理员端

## 3. `mini-program/`

这是 uni-app 小程序项目。

### 当前页面

- 登录页
- 律师首页
- 新建案件页
- 律师案件详情页
- 当事人进入页
- 当事人案件详情页

### 当前问题

- 还没有完整个人中心
- 还没有完整机构加入与审批状态流转
- 真实微信能力还未全面接入

## 4. `docs/`

这是中文文档中心。

### 当前作用

- 记录阶段清单
- 记录本地联调说明
- 记录部署说明
- 记录验收说明

### 后续建议

- 以后所有重要设计决策都写进文档
- 文档按“路线图、联调、部署、验收、问题清单”分类维护

## 二、角色流程

## 1. 平台超级管理员流程

### 目标流程

1. 登录平台后台
2. 查看所有租户
3. 查看平台收入和 AI 使用
4. 配置套餐和系统参数
5. 冻结异常租户

### 当前问题

- 这条流程还不存在

## 2. 机构管理员流程

### 目标流程

1. 注册机构
2. 创建租户
3. 邀请律师加入
4. 审批律师加入
5. 查看本所案件和统计
6. 购买订阅套餐

### 当前问题

- 机构创建和加入流程没有完全闭环
- 套餐与支付没接好

## 3. 律师流程

### 目标流程

1. 注册或加入机构
2. 登录 Web 或小程序
3. 创建案件
4. 维护案件状态
5. 上传材料
6. 邀请当事人
7. 查看材料与通知
8. 调用 AI 分析

### 当前问题

- 创建案件和查看材料已具备基础能力
- AI、支付、完整通知还缺失
- 律师加入机构流程不完整

## 4. 当事人流程

### 目标流程

1. 扫码或点击分享链接进入
2. 绑定手机号
3. 查看案件时间线
4. 上传材料
5. 接收律师通知

### 当前问题

- 本地演示版已经基本能跑通
- 但真机、正式分享、正式通知体验还没有完全补齐

## 三、从流程里能看出的关键问题

1. 平台层缺失
没有平台超级管理员，说明当前系统仍是“单机构视角”。

2. 租户层不完整
机构创建、加入、审批不完整，说明多租户只是有模型，还不是完整业务能力。

3. 产品核心价值还没落地
AI、计费、支付还没完成，说明系统目前主要是案件管理工具，不是完整商业产品。

4. 上线级基础设施还没完成
对象存储、异步任务、监控、告警、预发布都没真正形成体系。

## 四、建议你以后如何看项目进度

不要只看“页面能不能打开”，要按下面四层看：

1. 模型是否完整
2. 接口是否闭环
3. 页面是否可用
4. 部署和运维是否可上线

只要其中一层没完成，就不能算那个模块真正完成。
