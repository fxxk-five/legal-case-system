# 完整开发路线图

本文档基于你的完整产品目标整理，不再以“本地演示版”作为最终目标，而是把当前仓库视为中间版本。

## 一、总体目标

系统最终要覆盖六大阶段：

1. 基础架构与多租户
2. 核心业务逻辑（律师 Web 端）
3. 小程序端（律师 + 当事人）
4. AI 集成与计费系统
5. 平台超级管理员后台
6. 测试、部署与上线

## 二、优化后的执行原则

### 原则 1：先补底座，再补功能

如果多租户、权限、认证没有做扎实，后面越做越难改。

### 原则 2：先打通主链路，再加高级能力

主链路是：

- 律师注册或加入机构
- 律师创建案件
- 邀请当事人
- 当事人进入案件上传材料
- 律师处理材料

AI、支付、统计都应该建立在这条主链路稳定以后。

### 原则 3：演示版能力不能冒充正式版能力

mock 登录、后端中转上传、本地文件存储、默认租户逻辑，都只能算阶段性方案。

## 三、六阶段实施拆解

## 阶段一：基础架构与多租户

### 目标

把系统从“单租户演示”改成“真正可扩展的多租户底座”。

### 关键任务

- 重构租户创建流程
- 增加个人租户、机构租户、加入机构申请
- 增加审批流
- 在 token 中标准化写入 `tenant_id`、`user_id`、`role`、`is_tenant_admin`
- 建立数据库会话级租户上下文
- 建立应用层和数据库层双重权限隔离
- 完成 RLS 策略
- 补齐租户、用户、案件、文件、邀请等核心模型字段

### 验收标准

- 不同租户数据不能相互访问
- 律师加入机构需要审批
- 租户管理员能管理本机构成员
- 所有业务查询默认带租户约束

## 阶段二：核心业务逻辑

### 目标

完成律师 Web 端主工作台。

### 关键任务

- 案件看板
- 案件详情页增强
- 文件上传、预览、下载
- 截止日期、进度状态、负责人管理
- 通知中心
- 机构管理员后台
- 订阅信息和基础计费入口

### 验收标准

- 律师能高频使用 Web 完成日常案件管理
- 机构管理员能管律师、看统计、看订阅
- 文件权限与案件权限一致

## 阶段三：小程序端

### 目标

让律师和当事人都能在小程序完成核心动作。

### 关键任务

- 微信登录正式化
- 个体律师注册
- 律师加入机构
- 当事人扫码进入案件
- 当事人查看时间线并上传材料
- 律师查看案件、邀请当事人、查看通知
- 个人中心
- 分享、二维码、保存相册

### 验收标准

- 律师可在小程序查看案件
- 当事人能从邀请入口直接进入对应案件
- 上传材料后 Web 和小程序都能看到

## 阶段四：AI 集成与计费

### 目标

把系统的核心差异化价值真正做出来。

### 关键任务

- 接入真实大模型
- 设计 AI 抽象层
- Celery + Redis 异步任务
- 三大法宝：案件总结、诉讼策略、红蓝对抗
- 个人律师按次扣费
- 机构律师按套餐和配额计费
- AI 调用日志和统计

### 验收标准

- AI 调用不阻塞主接口
- 调用有成功失败记录
- 计费扣减可追踪、可回滚、可审计

## 阶段五：平台超级管理员后台

### 目标

支持平台运营，而不只是单一律所使用。

### 关键任务

- 独立超级管理员登录
- 租户管理
- 用户管理
- 套餐管理
- 系统参数管理
- 平台统计报表
- 审计日志

### 验收标准

- 可以从平台视角管理所有租户与套餐
- 可以看收入、调用量、增长趋势

## 阶段六：测试、部署与上线

### 目标

从开发版过渡到可交付上线版。

### 关键任务

- 自动化测试和集成测试
- 安全审计
- 预发布环境
- Docker Compose / Nginx / HTTPS
- 监控、告警、备份
- 腾讯云或阿里云部署
- 微信审核和上线资料准备

### 验收标准

- 有预发布验证流程
- 有数据库备份和恢复方案
- 有基础监控和错误告警

## 四、建议的近期冲刺顺序

当前不要直接做阶段四和阶段五，建议按下面顺序冲刺：

1. 阶段一缺口补齐
2. 阶段二核心页面补齐
3. 阶段三角色流程补齐
4. 阶段四 AI 与支付
5. 阶段五平台后台
6. 阶段六上线加固

## 五、你需要准备的外部资源

- 微信小程序 AppID / AppSecret
- 微信支付商户号
- 腾讯云服务器
- 域名和备案
- 腾讯云 COS 或阿里云 OSS
- 至少 1 到 2 位真实律师试用反馈
