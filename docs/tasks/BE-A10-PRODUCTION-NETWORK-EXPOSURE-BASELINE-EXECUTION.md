# BE-A10 执行单：生产端口与网络暴露基线

## 1. 任务目标
- 将生产网络暴露面收敛到最小集合，默认仅暴露 `80/443`。
- 避免数据库、缓存、应用内部端口直接暴露公网。
- 形成可直接落地的 `docker-compose + nginx + 安全组` 约束规范。

## 2. 范围
### 2.1 In Scope
- `docker-compose.yml` 端口暴露基线。
- `deploy/nginx/nginx.conf` 入口策略基线。
- 云服务器安全组与网络拓扑建议。
- 发布、回滚中的端口检查要求。

### 2.2 Out of Scope
- 跨区域容灾网络架构。
- 服务网格与零信任体系改造。

## 3. 现状基线（代码事实）
- 当前 `docker-compose.yml` 暴露：
  - `5432:5432`（PostgreSQL）
  - `6379:6379`（Redis）
  - `8000:8000`（backend）
  - `8080:80`（web-frontend）
  - `80:80`（nginx）
- 当前 `nginx.conf` 仅监听 `80`，无 `443` 与 TLS 配置。
- `docs/production-deployment.md` 已标注“生产应关闭直出端口”，但尚未形成执行细则。

## 4. 目标网络拓扑
- 公网入口：`nginx`（`80/443`）。
- 内网服务：
  - `backend:8000`（仅容器网络可见）
  - `web-frontend:80`（仅容器网络可见）
  - `postgres:5432`（仅容器网络可见）
  - `redis:6379`（仅容器网络可见）
- 原则：除 `nginx` 外，其他服务不做 host 端口映射。

## 5. 端口基线规范

## 5.1 Compose 层
- 生产环境禁止以下 `ports` 直出：
  - postgres 5432
  - redis 6379
  - backend 8000
  - web-frontend 8080
- 使用 `expose` 或默认容器网络通信替代。
- 建议通过 `docker-compose.prod.yml` 与本地开发配置分离。

## 5.2 主机/安全组层
- 允许入站：
  - `22`（运维 IP 白名单）
  - `80`（可选，后续跳转到 443）
  - `443`（HTTPS）
- 默认拒绝：
  - `5432`、`6379`、`8000`、`8080`

## 5.3 Nginx 层
- 补齐 TLS：
  - `80 -> 443` 重定向
  - `443` 挂载证书
- 增加基础安全头与请求体大小限制（按文件上传策略对齐）。

## 7. 验收标准（DoD）
- 生产环境公网扫描仅发现 `80/443`（和可选 `22`）。
- backend/postgres/redis 端口无法从公网直接访问。
- HTTPS 可用，小程序合法域名配置可通过校验。
- 发布与回滚均包含端口暴露检查步骤。

## 8. 风险与默认策略
- 风险：误关端口导致联调/运维受阻。  
  默认策略：本地与生产配置分离，生产执行前进行灰度验证。
- 风险：只改 compose 未改安全组，仍可被访问。  
  默认策略：compose 与云安全组双重收敛，缺一不可。
- 风险：HTTPS 未完成导致小程序域名校验失败。  
  默认策略：上线前必须完成证书配置与域名验证。

## 9. 回滚策略
1. 若收敛后出现服务不可达，临时恢复旧 compose 映射并限制来源 IP。
2. 记录回滚窗口，完成根因修复后再次收敛。
3. 回滚期间不得长期开放数据库与缓存公网端口。
