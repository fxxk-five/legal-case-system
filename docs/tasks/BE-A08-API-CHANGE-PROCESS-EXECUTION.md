# BE-A08 执行单：API 变更流程规范

## 1. 任务目标
- 建立可追溯、可回滚的 API 变更流程，确保契约变更不再“代码先行、文档滞后”。
- 将 `docs/API-CONTRACTS.md` 固化为单一真源（SSOT），并定义变更审批与发布窗口。
- 降低 Web/小程序联调中的路径、方法、字段漂移风险。

## 2. 范围
### 2.1 In Scope
- API 变更分级（破坏性/非破坏性/文档修正）。
- 变更流程（提案、评审、实现、联调、发布、回滚）。
- 契约文档模板与字段级变更记录模板。
- 兼容窗口与废弃路径下线机制。

### 2.2 Out of Scope
- CI 平台替换或组织流程重构。
- 网关产品升级。

## 3. 现状基线（代码事实）
- 目前已有 `docs/API-CONTRACTS.md`，但前端调用与契约存在多处偏差（如 Web `CaseDetailView`、`LawyersView` 旧路径）。
- API 变更未形成统一模板与版本窗口记录，导致“实现态/文档态”可能短期分叉。

## 4. 变更分级定义
- L0 文档修订：不改接口，仅修正描述。
- L1 非破坏性：新增字段/新增可选参数/新增接口，不影响旧调用。
- L2 兼容变更：同接口支持新旧参数双轨，含兼容窗口。
- L3 破坏性：路径、方法、必填参数、响应结构变化，必须走版本化或双轨。

## 5. 标准流程（必须执行）
1. 提案：创建变更单，填写“变更原因、影响范围、回滚方案、联调计划”。
2. 评审：后端 + Web + 小程序三方评审，确认契约与迁移路径。
3. 契约先行：先更新 `docs/API-CONTRACTS.md` 与相关执行单，再进入实现。
4. 实现：后端改造 + 必要前端接口对齐改造（禁止 UI 变更）。
5. 测试：单元/集成 + 双端联调回归。
6. 发布：记录上线时间、版本窗口、废弃路径公告。
7. 回滚：按预设步骤快速恢复旧契约行为。

## 7. 版本与兼容策略
- 现行版本固定：`/api/v1`。
- L3 变更默认策略：
  - 优先双轨兼容（旧路径 + 新路径并行）；
  - 若不能双轨，需给出明确迁移窗口和强提醒。
- 废弃机制：
  - `标记废弃 -> 双轨期 -> 下线`，每阶段有具体日期。

## 9. 验收标准（DoD）
- 任一 API 变更都能追溯到变更单与契约文档更新记录。
- 契约文档与代码状态一致，不存在“只改代码不改契约”。
- 破坏性变更具备兼容窗口与可执行回滚步骤。
- Web/小程序联调时可依据变更单快速定位影响接口。

## 10. 风险与默认策略
- 风险：流程执行成本上升导致绕流程。  
  默认策略：L0/L1 轻流程，L2/L3 强流程。
- 风险：双轨窗口过长导致技术债。  
  默认策略：默认 1~2 个迭代窗口，超期必须复审。
- 风险：多人并行修改契约冲突。  
  默认策略：以 CHANGE-ID 为单位串行合并契约章节。

## 11. 回滚策略
1. 发布前必须有“旧行为可恢复”验证记录。
2. 发现大面积联调失败时，优先启用旧路径兼容。
3. 回滚后在变更台账登记根因与修正计划。
