# 腾讯云 `CVM + COS` 商用订阅说明

> 文档版本：V1.2  
> 适用场景：当前阶段先以 `CVM + COS` 方式上线，后续逐步演进到多租户订阅、可扩容、可升级的云上架构

---

## 1. 结论

对于当前项目，推荐的首阶段商用架构是：

`1 台 CVM + Docker Compose + COS + 域名/SSL + 可选 CDN`

这条路线的目的不是一步做到最终云原生，而是先满足三个约束：

- 服务必须在云端独立启动和运行，不依赖开发机或办公室网络。
- 文件和报告必须存放在云端，不继续绑在单台主机磁盘上。
- 后续做多租户订阅、容量升级、运行面扩容时，不需要重做文件存储架构。

---

## 2. 为什么是 `CVM + COS`

### `CVM`

适合当前阶段快速上线现有 `Docker Compose` 结构：

- `nginx`
- `web-frontend`
- `backend`
- `ai-worker`
- `report-service`
- `postgres`
- `redis`

优点：

- 与现有仓库部署方式最接近，上线成本最低。
- 对 `report-service` 里的 Chromium / Puppeteer 更友好。
- 运维路径简单，适合首个律所和首批租户上线。

限制：

- 单机不是高可用。
- 数据库、缓存、worker、API 都在同一台主机时，扩容边界明显。
- 如果文件仍然放本地盘，后续迁移和多实例扩容会变得很别扭。

### `COS`

建议现在就接入，不要等多租户后再补。

原因：

- 把文件与报告从 `CVM` 本地磁盘拆出去，后续换机器、加机器、切 `CLB` 或切容器平台都不会重新搬存储模式。
- 你选择的是 `CVM 按流量计费`，附件下载、报告下载、预览如果继续走主机公网，会直接推高账单；文件流量交给 `COS` 更合理。
- 多租户订阅以后，可以按 `tenant_id` 管理对象前缀、存储用量、生命周期和归档策略。

## 3. 推荐订阅清单

### 3.1 首阶段必选资源

| 资源 | 是否必选 | 用途 | 订阅建议 |
| --- | --- | --- | --- |
| `VPC` / 子网 | 必选 | 云内网络隔离 | 单地域单 VPC 起步 |
| `CVM` | 必选 | 运行 `Docker Compose` 全套服务 | `2C4G` 起步，正式商用更建议 `4C8G` |
| 系统盘 / 数据盘 | 必选 | 系统、镜像、数据库、缓存、临时处理目录 | 建议 `SSD 100GB` 起步 |
| `COS Bucket` | 必选 | 案件附件、报告、导出文件 | 私有读写，标准存储 |
| 域名 + `DNSPod` | 必选 | 统一访问入口 | 至少准备主域名和 `api` 子域名 |
| `SSL` 证书 | 必选 | HTTPS | 首阶段可用免费证书 |
| 安全组 | 必选 | 最小暴露面 | 仅开放 `80/443/22`，`22` 建议白名单 |
| 云监控 / 告警 | 必选 | 基础可运维 | 监控 CPU、内存、磁盘、出网流量 |
| 快照 / 备份 | 必选 | 主机恢复 | 至少保留主机快照与数据库备份策略 |

### 3.2 首阶段建议资源

| 资源 | 作用 | 何时建议购买 |
| --- | --- | --- |
| `CDN` | 加速静态资源、附件下载、报告下载 | 文件下载和跨地域访问明显增多时 |
| `TencentDB PostgreSQL` | 把数据库从单机拆出去 | 开始做多租户正式商用时 |
| `TencentDB Redis` | 提升缓存与队列稳定性 | 并发和会话数上升时 |
| 第二台 `CVM` | 为 `CLB + 多实例` 做准备 | 访问量或可用性要求提高时 |
| `CLB` | 负载均衡 | 需要多实例或切流时 |

### 3.3 后续扩容阶段资源

| 资源 | 用途 | 适用时机 |
| --- | --- | --- |
| `TDMQ/CMQ` | 替代 `DB Queue` | AI 任务量和重试治理复杂度上来时 |
| `WAF` | 公网安全治理 | 正式商用、暴露面变大时 |
| `TCR` | 镜像仓库 | 需要标准化发布流程时 |
| `TKE` | 容器编排 | 多实例、滚动发布、自动扩缩时 |
| `KMS / Secrets Manager` | 密钥治理 | 环境和密钥数量变多时 |

## 4. 按流量计费模式下，为什么更要上 `COS`

如果 `CVM` 采用按流量计费，主机公网流量应尽量只承担：

- API 请求
- 登录鉴权
- 页面动态数据
- 少量管理后台静态资源

不应该长期承担：

- 附件原文件下载
- 报告 PDF 下载
- 图片预览
- 高频静态资源分发

推荐分工：

- `CVM`：处理业务逻辑、鉴权、上传签名、下载签名、AI 编排。
- `COS`：存附件、存报告、存导出文件。
- `CDN`：文件访问多起来后再加，继续降低 `CVM` 出网流量。

## 5. 对多租户订阅和后续扩容的意义

`COS` 对多租户订阅最直接的价值有三点：

- 便于按租户前缀做对象隔离，例如 `tenant_{id}/case_{id}/...`
- 便于统计每个租户的存储量、下载量和生命周期策略
- 便于后续把运行面从单台 `CVM` 扩到多台 `CVM` 或 `TKE`，而不用重做文件搬迁

推荐演进路径：

1. `Phase A`：`1 台 CVM + COS`
2. `Phase B`：`CVM + COS + 托管 PostgreSQL/Redis`
3. `Phase C`：`CLB + 2 台以上 CVM + COS`
4. `Phase D`：`TCR + TKE + COS + 托管数据面`

这条路径的关键是：`文件存储模式从第一阶段起就保持不变`。

## 6. 当前仓库与该方案的差距

当前仓库已经完成首轮 `CVM + COS` 存储链路改造，旧的“本地存储链路”描述已不再成立。当前状态应更新为：

- 后端已同时支持 `local` 与 `cos` 两种存储后端，生产目标后端为 `COS`。
- 文件直传链路已改为“上传授权 -> 前端直传 `COS` -> 完成回写确认 -> 元数据入库”。
- 直传对象会先进入 `tenant_{tenant_id}/case_{case_id}/_pending/`，确认成功后再晋级到 `tenant_{tenant_id}/case_{case_id}/files/`。
- 报告文件已按 `tenant_{tenant_id}/case_{case_id}/reports/report-{scope}-{timestamp}.pdf` 写入对象存储，并优先返回签名下载地址。
- 文件删除已支持按策略归档到 `_retained/deleted_at=.../`，避免生产环境直接静默丢失对象。
- `docker-compose.prod.yml` 已移除生产对本地主机文件卷作为“证据/报告真源”的依赖，生产真源应为 `COS`。
- 生产 compose 已切换为 `deploy/backend.env.tencent.prod.example + deploy/backend.env.tencent.prod`，不再直接依赖 `backend/.env`。
- 已补充 `dev / staging / production` 环境与密钥治理映射文档，首阶段受控 env 文件、后续 `Secrets Manager / KMS` 的口径已收口。
- `ai-worker` 已补充心跳文件、容器健康检查、延迟重试、陈旧任务恢复与死信护栏。
- 已补充单机生产的备份、恢复、日志轮转模板与运行手册说明。
- 已补充二阶段演进蓝图，覆盖 `TencentDB PostgreSQL/Redis`、`CLB + 多 CVM`、`TDMQ/CMQ`、`TCR + TKE`、`CDN/WAF/CLB` 与分波切流方案。

当前真正剩余的差距不在“代码是否还是本地存储”，而在云侧落地与正式联调：

- `COS Bucket` 生命周期、CORS、最小权限 `CAM` 仍需按生产环境配置完成。
- 域名、`SSL`、`DNSPod`、小程序合法域名、正式公网入口还未全部落地。
- 正式云域名上的端到端验收尚未执行，因此还不能把“云端已验收通过”打勾。

## 13. 标准任务卡规范

### 13.1 任务编号规则

统一编号格式：

`[团队]-[阶段]-[序号]`

示例：

- `OPS-P0-001`
- `BE-P1-003`
- `FEWEB-P2-002`
- `FEMINI-P2-004`
- `QA-P3-001`

### 13.2 标准任务卡字段

每一条任务必须包含以下字段：

| 字段 | 是否必填 | 说明 |
| --- | --- | --- |
| 任务编号 | 必填 | 唯一标识 |
| 任务标题 | 必填 | 目标必须可判断是否完成 |
| 所属阶段 | 必填 | `P0/P1/P2/P3/P4` |
| 执行负责人 `R` | 必填 | 单一责任人 |
| 结果负责人 `A` | 必填 | 拍板人与验收确认人 |
| 前置依赖 | 必填 | 没有就写“无” |
| 输入物 | 必填 | 依赖哪些资源、接口、配置 |
| 执行步骤 | 必填 | 具体到可照做 |
| 输出物 | 必填 | 文档、配置、代码、截图、记录 |
| 自检项 | 必填 | 执行人先自检 |
| 交叉复核项 | 必填 | 由协作方或 QA 复核 |
| 回滚方式 | 必填 | 失败时如何撤回 |
| 风险等级 | 必填 | `低/中/高` |
| 计划开始时间 | 必填 | 时间窗 |
| 计划完成时间 | 必填 | 时间窗 |
| 当前状态 | 必填 | 使用统一状态机 |

### 13.3 标准状态机

任务状态统一为：

`todo -> ready -> in_progress -> self_check -> peer_review -> ready_for_release -> released -> accepted -> archived`

异常状态：

`blocked`

状态切换规则：

- [ ] 未确认前置依赖，不允许从 `todo` 进入 `ready`。
- [ ] 未完成自检，不允许进入 `peer_review`。
- [ ] 未完成复核，不允许进入 `ready_for_release`。
- [ ] 未经过发布，不允许标记为 `accepted`。
- [ ] 被阻塞时必须填写阻塞原因、责任归属、预计解除时间。

### 13.4 单任务粒度规则

- [ ] 单任务建议控制在 `0.5 天 ~ 2 天` 内完成。
- [ ] 超过 2 天的任务必须拆成子任务。
- [ ] 一个任务只交付一个明确目标，不允许把“开发 + 测试 + 上线 + 验收”混成一个任务。
- [ ] 高风险任务必须独立成卡，例如：
  - 域名切换
  - 主机重启验证
  - 数据迁移
  - 文件存储切换
